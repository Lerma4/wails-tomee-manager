## Project Overview

**tomee-manager** is a desktop application for managing a local Apache TomEE server. It allows users to configure TomEE paths/ports, manage WAR artifact deployments, and start/stop/restart the server with live log streaming. Built with [Wails v2](https://wails.io/) (Go backend + React/TypeScript frontend).

## Commands

### Development
```bash
wails dev          # Run in live development mode (starts both Go backend and Vite dev server)
```

### Build
```bash
wails build        # Build production binary (output: tomee-mananger.exe)
```

### Verify
```bash
go build ./...     # Quick compile check (no tests exist yet)
```

### Frontend only (from frontend/ directory)
```bash
npm install        # Install frontend dependencies
npm run dev        # Vite dev server only
npm run build      # TypeScript check + Vite production build
```

## Architecture

### Backend (Go)

- **main.go** ŌĆö Wails app entry point. Initializes services, binds them to the frontend, and embeds `frontend/dist` via `//go:embed`.
- **app.go** ŌĆö `App` struct with Wails lifecycle (`startup`) and native dialog helpers (`SelectDirectory`, `SelectWarFile`).
- **backend/service/storage.go** ŌĆö `StorageService`: SQLite persistence (via `modernc.org/sqlite`, pure Go, no CGO). Database stored at `os.UserConfigDir()/tomee-manager/data.db`. Manages `config` (singleton row) and `wars` tables.
- **backend/service/tomee.go** ŌĆö `TomEEService`: Start/Stop/Restart TomEE via `catalina.bat`/`catalina.sh`. Modifies `server.xml` ports using `etree` XML library. Streams stdout/stderr logs to frontend via Wails events (`tomee-log`).
- **backend/service/war.go** ŌĆö `WarService`: Copies enabled WAR artifacts to TomEE's `webapps/` directory.
- **backend/model/config.go** ŌĆö Data models: `Config` (TomEE path + ports) and `WarArtifact` (source path, dest name, enabled flag).

All services are bound to the frontend in `main.go` ŌĆö their exported methods become callable from TypeScript via auto-generated bindings in `frontend/wailsjs/` (gitignored, regenerated by `wails dev`/`wails build`).

### Frontend (React + TypeScript)

- **Tech stack**: React 18, React Router (HashRouter), Tailwind CSS v4, DaisyUI v5, Vite
- **frontend/src/App.tsx** ŌĆö Router with three routes: `/` (Dashboard), `/wars` (WAR Manager), `/config` (Configuration)
- **Pages**:
  - `Dashboard.tsx` ŌĆö Start/Stop/Restart buttons + live log viewer (subscribes to `tomee-log` Wails event)
  - `WarManager.tsx` ŌĆö CRUD table for WAR artifacts + "Deploy All" action
  - `Configuration.tsx` ŌĆö Form for TomEE path and port settings
- **Go bindings** are imported from `../../wailsjs/go/service/` and `../../wailsjs/go/main/` (e.g., `Start()`, `LoadConfig()`, `DeployAll()`)

### Key Patterns

- **TomEEService concurrency**: Public methods (`Start`, `Stop`, `Restart`) acquire `s.mu`; internal `startLocked`/`stopLocked` assume caller holds the lock. A background goroutine clears `s.process` on natural exit. Always set `CATALINA_HOME`/`CATALINA_BASE` env vars on both start and shutdown commands.
- **Go ŌåÆ Frontend communication**: Wails `EventsEmit` for log streaming; frontend listens via `EventsOn` from `wailsjs/runtime/runtime`
- **Frontend ŌåÆ Go communication**: Direct function calls via Wails-generated TypeScript bindings (no REST API)
- **Single-row config**: The `config` table enforces `id = 1` via CHECK constraint
- **Cross-platform**: TomEE service selects `catalina.bat` vs `catalina.sh` based on `runtime.GOOS`

### Versioning

- **Source of truth**: `frontend/package.json` ŌåÆ campo `"version"` (es. `"0.1.0"`)
- **Build-time injection**: `vite.config.ts` espone `__APP_VERSION__` via `define` ŌĆö usata dal componente `Footer.tsx`
- **Git tag convention**: i tag devono corrispondere alla versione in `package.json` (es. `v0.1.0`)
- Quando si rilascia una nuova versione: aggiornare `"version"` in `package.json` e taggare il commit (`git tag v<version>`)
